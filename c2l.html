Code2LoRA: Repository-Conditioned
LoRA Generation for Code Completion

A hypernetwork that maps repository-level code embeddings to LoRA adapters, enabling repository-specific code completion without per-repo fine-tuning.

Abstract

Large language models for code excel at general programming tasks but struggle with repository-specific patterns — custom APIs, project conventions, and internal dependencies that are invisible at inference time. Fine-tuning a separate adapter per repository is effective but impractical at scale.

We introduce Code2LoRA, a hypernetwork that maps a repository's code embedding to a full set of LoRA adapter weights in a single forward pass. Combined with Dynamic Repository Context (DRC) — an oracle retrieval mechanism that injects the most relevant source files into the prompt — Code2LoRA enables repository-aware code completion without per-repo training. We evaluate on a benchmark of 560 open-source Python repositories with over 73K assertion-completion tasks, demonstrating significant improvements over pretrained, per-repo LoRA, and full fine-tuning baselines.

Method

1

Repo Embedding

Chunk & embed all files, then pool to a single repo-level vector

2

Hypernetwork

Map the embedding to LoRA weight matrices (A, B) for each target layer

3

DRC Retrieval

Retrieve relevant source context and prepend it to the input

4

Inference

Inject LoRA into Qwen and predict assertion completions

Figure 1: Code2LoRA architecture. Repository files are embedded and aggregated into a single vector, processed by a GRU encoder, and mapped through a PAW-style generator to produce LoRA adapter weights for all 196 injection points in the frozen Qwen2.5-Coder. In parallel, Dynamic Repository Context (DRC) retrieves relevant source definitions via AST import resolution and prepends them to the test prompt.

Dataset

We curate a benchmark from 560 public Python repositories that use pytest, filtered by activity, size, and star count. From test files we extract assertion-completion pairs where the model must predict the expected value after an assert statement.

  • 73,267 QnA pairs across 560 repos (30–200 per repo)
  • Train / Val / Test split: 447 / 55 / 58 repos (cross-repo evaluation)
  • Assertions span numeric literals, strings, variables, collections, function calls, and more
  • Each pair includes an optional DRC oracle context retrieved from the source repository

Results

Cross-repo test set evaluation (58 held-out repositories, 7,620 assertion pairs):

Method Exact Match (%) Edit Similarity
Qwen (pretrained) 34.6 0.530
Code2LoRA (hypernetwork) 58.0 0.792

Additional baselines (per-repo LoRA, FFT, RAG, Text2LoRA) and ablations are reported in the paper.
